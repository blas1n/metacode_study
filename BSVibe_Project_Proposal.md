# BSVibe 기획서

## 1. 서비스 개요

### 1.1 프로젝트명 및 개요

**프로젝트명:** BSVibe

**개요:** BSVibe는 1인 창업자와 소규모 팀이 AI에게 제품 작업을 맡기고, 과거 결정과 피드백을 RAG로 재사용해 더 일관된 결과를 얻도록 돕는 AI 작업 운영 서비스입니다.

1인 창업자와 소규모 팀이 여러 제품 작업을 AI에게 맡기고, 필요한 결정과 검증만 직접 관리할 수 있도록 돕는 AI 작업 운영 서비스입니다.

사용자는 자연어로 작업을 지시하고, AI는 작업을 실행 가능한 단위로 나누어 처리합니다. 작업 중 중요한 선택지가 생기면 사용자에게 결정을 요청하고, 완료된 결과는 테스트, 파일, 링크, 로그 등 확인 가능한 근거와 함께 제공합니다.

작업 중 발생하는 사용자의 결정, 피드백, 작업 결과가 지식으로 계속 쌓이고, 다음 작업에서 다시 검색되어 활용되는 dynamic RAG 구조입니다. 이를 통해 AI가 같은 실수를 반복하지 않고, 사용자의 선호와 기준에 점점 맞춰지는 것을 목표로 합니다.

https://bsvibe.dev

### 1.2 설계 영감

BSVibe의 지식 구조는 다음 세 가지 아이디어에서 영감을 받았습니다.

- **Digital Garden:** 지식을 완성된 문서가 아니라 계속 자라고 연결되는 note로 다룹니다. 처음에는 작은 기록으로 시작하고, 반복적으로 중요해진 내용만 더 강한 지식으로 발전시킵니다.
- **Palantir Ontology:** 단순 문서 저장이 아니라 사람, 제품, 작업, 결정, 검증 결과 같은 업무 객체 사이의 관계를 표현하는 것을 목표로 합니다. 사용자는 흩어진 데이터를 보는 것이 아니라, 실제 작업 맥락 안에서 연결된 지식을 확인합니다.
- **Karpathy Wiki:** 개인의 생각, 결정, 작업 기록이 장기적으로 축적되는 개인 지식 베이스를 지향합니다. 다만 단순 위키가 아니라, AI가 다음 작업에서 직접 참고할 수 있는 실행용 메모리로 활용합니다.

## 2. 타겟 유저 및 해결하려는 문제

AI 코딩 도구와 자동화 도구는 빠르게 발전하고 있지만, 실제 제품 작업에서는 여전히 사용자가 많은 부분을 직접 관리해야 합니다.

- 여러 AI 작업을 동시에 진행하면 상태를 계속 확인해야 합니다.
- AI가 완료했다고 말해도 실제 검증 결과를 따로 확인해야 합니다.
- 사용자의 선호, 결정, 과거 피드백이 다음 작업에 자동으로 반영되지 않습니다.
- 작업 지시, 결과물, 회고, 문서가 여러 도구에 흩어져 재사용하기 어렵습니다.
- 같은 실수나 같은 질문이 반복되어 사용자의 관리 부담이 줄어들지 않습니다.

저는 이 문제를 "AI 실행" 문제가 아니라 "AI 작업 운영과 학습" 문제로 봤습니다.

### 2.1 타겟 유저

1차 타겟은 기술 이해도가 있는 1인 창업자 또는 소규모 SaaS 빌더입니다.

사용자는 제품 기획, 개발, 배포, 운영을 직접 관리할 수 있지만, 모든 작업을 매번 직접 지시하고 검수하는 데 많은 시간을 쓰고 싶어 하지 않습니다.

### 2.2 예상 사용 상황

- 신규 기능의 기획서와 작업 목록을 만들고 싶을 때
- GitHub 이슈를 바탕으로 코드 수정과 테스트를 맡기고 싶을 때
- 여러 제품의 진행 상황을 한 화면에서 보고 싶을 때
- AI가 만든 결과물을 근거와 함께 검토하고 싶을 때
- 이전 결정과 사용자 피드백을 다음 작업에 자동 반영하고 싶을 때

## 3. 실제 유저 시나리오

### 3.1 시나리오: 과거 피드백을 반영한 작업 답변 받기

1. 사용자가 Dashboard에서 진행 중인 작업과 확인이 필요한 항목을 확인합니다.
2. New Request 화면에 "결제 설정 화면의 모바일 UX를 점검하고 수정 방향을 제안해 주세요."라고 입력합니다.
3. 서비스는 요청을 분석하고, 과거 작업 기록과 문서에서 "모바일 화면은 스크린샷 기반으로 확인해야 한다", "테스트 없는 완료는 허용하지 않는다" 같은 사용자 피드백을 검색합니다.
4. AI는 검색된 컨텍스트를 바탕으로 작업 계획을 세우고, 관련 파일과 문서를 확인합니다.
5. 작업 중 "디자인 수정안을 먼저 제안할지, 바로 코드 수정까지 진행할지"처럼 사용자 판단이 필요한 부분이 생기면 Decisions 화면에 질문을 올립니다.
6. 모바일에서 "먼저 수정안을 제안해 주세요"를 선택합니다.
7. AI는 사용자의 결정을 반영해 수정 제안, 근거, 예상 영향, 추가로 필요한 검증 항목을 생성합니다.
8. 사용자는 Delivery Report에서 최종 답변, 참고한 문서, 과거 피드백, 검증 체크리스트를 함께 확인합니다.
9. 이번 작업에서 나온 새 결정과 피드백은 다시 지식으로 저장되어 다음 요청에서 재사용됩니다.

## 4. 서비스 동작 흐름

BSVibe의 전체 흐름은 "요청을 받는 단계 → 관련 지식을 찾는 단계 → AI가 작업하는 단계 → 사용자가 필요한 결정을 하는 단계 → 결과를 검증하고 저장하는 단계"로 구성됩니다.

| 단계 | 설명 | 산출물 |
|---|---|---|
| 1. 입력 수집 | 사용자의 직접 지시 또는 외부 도구의 이벤트가 들어옵니다. | 작업 요청 |
| 2. 요청 해석 | 요청의 의도, 관련 제품, 필요한 산출물을 해석합니다. | 작업 계획 |
| 3. 지식 검색 | 과거 결정, 사용자 선호, 반복 패턴을 검색합니다. | 작업 컨텍스트 |
| 4. 작업 실행 | AI가 도구와 문서를 사용해 작업을 수행합니다. | 중간 산출물 |
| 5. 결정 요청 | 사용자의 판단이 필요한 선택지를 분리합니다. | 승인/반려 요청 |
| 6. 검증 | 테스트, 파일, 로그, 링크 등으로 결과를 확인합니다. | 검증 결과 |
| 7. 전달 | 검증된 결과물을 사용자에게 보여주거나 외부 도구로 전송합니다. | 최종 산출물 |
| 8. 학습 | 작업 결과, 결정, 피드백을 지식으로 저장합니다. | 재사용 가능한 지식 |

## 5. RAG 적용 방식

BSVibe의 RAG는 고정된 문서 묶음을 검색하는 일반적인 Q&A 구조와 다릅니다. 사용자가 서비스를 사용할수록 데이터가 쌓이고, 그 데이터가 다음 작업의 컨텍스트로 다시 사용됩니다.

### 5.1 지식이 생성되는 경로

| 입력 경로 | 예시 | 저장되는 지식 |
|---|---|---|
| 사용자 지시 | "모바일 화면부터 우선 확인해 주세요." | 우선순위, 표현 방식, 작업 맥락 |
| 사용자 결정 | "보안 위험이 낮은 A안을 선택하겠습니다." | 판단 기준, 선호, 의사결정 이유 |
| 작업 결과 | PR, 문서, 테스트 결과, 배포 로그 | 검증된 산출물과 증거 |
| 사용자 피드백 | "테스트가 없으면 완료로 보지 않겠습니다." | 반복 방지 규칙 |
| 외부 도구 입력 | GitHub issue, 문서, 메시지 등 | 외부 업무 맥락 |
| 기존 문서 import | README, PRD, 운영 문서, 회고 | 초기 지식 기반 |

### 5.2 저장 구조

모든 작업 기록을 곧바로 강한 규칙으로 사용하지 않습니다. 먼저 작업 결과와 사용자 결정이 지식 note로 저장되고, 반복적으로 등장한 패턴만 중요한 개념으로 승격됩니다.

| 저장 단위 | 설명 | 예시 |
|---|---|---|
| 원본 작업 기록 | 사용자의 요청, 작업 결과, 검증 결과, 결정 내역처럼 나중에 추적해야 하는 원본 기록입니다. | 작업 요청, 실행 로그, 검증 결과 |
| 지식 note | 작업이 끝난 뒤 재사용할 수 있도록 요약된 지식 기록입니다. | "모바일 화면은 스크린샷 기반으로 확인해야 함" |
| 반복 패턴 | 여러 작업에서 반복되어 중요해진 기준이나 선호입니다. | "테스트 없는 완료는 허용하지 않음" |
| 사용자 결정 기록 | 사용자가 특정 상황에서 어떤 결정을 했는지 남긴 기록입니다. | "보안 위험이 낮은 A안을 선택" |
| 지식 연결 | 반복 패턴과 작업 기록 사이의 관계를 보여주는 연결 정보입니다. | `mobile-ui`와 `screenshot-check`의 반복 연결 |

핵심은 원본 기록과 반복 패턴을 분리하는 것입니다. 한 번 나온 AI 답변이나 작업 로그를 바로 규칙처럼 사용하면 오류가 누적될 수 있기 때문입니다. 따라서 반복적으로 확인된 패턴만 다음 작업의 강한 기준으로 사용합니다.

### 5.3 검색을 위한 데이터 가공

원본 지식은 사람이 읽고 수정할 수 있도록 Markdown note로 저장합니다. 하지만 검색할 때마다 모든 Markdown 파일을 처음부터 LLM에게 넘기지는 않습니다. 검색 목적에 맞게 다음과 같이 가공합니다.

| 가공 단계 | 설명 | 검색에서의 역할 |
|---|---|---|
| Markdown frontmatter 추출 | note의 태그, 생성 시점, 작업 id, 결정 여부 같은 메타데이터를 분리합니다. | 특정 종류의 note만 빠르게 찾습니다. |
| 태그 정규화 | 표현이 조금 다른 단어를 비교할 수 있도록 소문자, 하이픈 형태의 태그로 정리합니다. | 반복 패턴 후보를 찾습니다. |
| 반복 패턴 승격 | 여러 작업에서 반복적으로 등장한 태그나 피드백을 중요한 개념으로 승격합니다. | 다음 작업의 강한 기준으로 사용합니다. |
| 결정 기록 구조화 | 사용자의 질문, 선택지, 답변, 이유를 구조화된 필드로 저장합니다. | 비슷한 상황에서 과거 결정을 검색합니다. |
| 지식 연결 생성 | 같은 작업이나 note에서 함께 등장한 개념 사이의 연결을 만듭니다. | Knowledge 화면에서 관계를 보여주고 맥락 검색을 보강합니다. |

### 5.4 검색 구조와 사용하는 저장소

현재 작업에 직접 사용되는 검색은 크게 두 가지를 중심으로 합니다.

1. **반복 패턴 검색:** 이전 작업에서 여러 번 확인되어 중요한 기준으로 승격된 사용자 선호와 규칙을 찾습니다.
2. **과거 결정 검색:** 비슷한 상황에서 사용자가 이미 어떤 결정을 했는지 찾습니다.

예를 들어 사용자가 과거에 "테스트가 없으면 완료로 보지 않는다"고 여러 번 피드백했다면, 이 내용은 다음 작업의 검증 기준으로 검색됩니다. 또한 비슷한 선택지가 다시 나오면 과거에 사용자가 선택했던 답변을 참고할 수 있습니다.

현재 MVP의 핵심 검색은 Markdown 파일을 기반으로 한 인메모리 인덱스를 사용합니다. 즉, 반복 패턴과 과거 결정은 Markdown에 저장된 메타데이터와 태그를 읽어 검색합니다. 별도의 벡터 DB만으로 모든 것을 해결하는 구조가 아니라, 사용자가 읽을 수 있는 Markdown을 source of truth로 두고 필요한 색인만 파생해서 사용합니다.

| 검색 대상 | 사용하는 저장소/인덱스 | 설명 |
|---|---|---|
| 반복 패턴 | Markdown 기반 인메모리 인덱스 | 승격된 개념 note를 읽어 현재 작업 신호와 맞는 기준을 찾습니다. |
| 과거 결정 | Markdown frontmatter 검색 | 결정 note의 질문, 답변, 의도 필드를 읽고 현재 요청과 겹치는 토큰을 찾습니다. |
| 의미 기반 note 검색 | PostgreSQL + pgvector | note 제목과 본문을 embedding으로 변환해 `vector` 타입으로 저장하고, cosine distance로 유사 note를 찾습니다. |
| 지식 연결 | Markdown에서 파생한 graph 데이터 | 개념과 작업 기록 사이의 연결을 재구성해 관계를 보여줍니다. |

따라서 현재 기획의 핵심은 "모든 문서를 많이 검색하는 것"보다 "반복된 결정과 검증 기준을 정확히 재사용하는 것"입니다. 의미 기반 벡터 검색은 보조 검색 또는 확장 기능으로 사용합니다.

### 5.5 RAG가 사용되는 위치

| 위치 | RAG 사용 목적 |
|---|---|
| 요청 해석 | 사용자의 요청이 어떤 작업 유형인지 판단합니다. |
| 작업 시작 | 관련 과거 결정과 사용자 선호를 초기 컨텍스트로 제공합니다. |
| 작업 중 검색 | 필요한 반복 패턴이나 과거 결정을 찾아 작업에 반영합니다. |
| 검증 단계 | 과거 피드백을 바탕으로 이번 작업에서 반드시 확인할 기준을 보강합니다. |
| 단순 질문 응답 | 실행 없이 저장된 지식만으로 답변할 수 있는 질문에 답합니다. |
| 결정 요청 | 이미 비슷한 결정을 한 적이 있으면 이전 결정을 참고하도록 제안합니다. |

### 5.6 안전 장치

- 검색 결과에는 출처 등의 메타데이터를 함께 저장해 추적 가능하게 합니다.
- 반복 패턴은 충분한 근거가 쌓인 뒤 규칙처럼 사용합니다.
- 외부로 전송되는 결과물은 필요 시 승인 큐를 거칩니다.
- 완료 판단은 AI의 자기평가보다 테스트, 실행 결과, 파일, 링크 같은 결정적 증거를 사용합니다.

## 6. 사용할 데이터셋의 특성

### 6.1 데이터 출처 및 확보 방법

BSVibe의 데이터는 공개 데이터셋을 한 번 다운로드해서 사용하는 방식이 아니라, 사용자가 서비스를 쓰는 과정에서 동적으로 생성됩니다.

주요 데이터 출처는 다음과 같습니다.

| 출처 | 확보 방법 | 설명 |
|---|---|---|
| 사용자 입력 | 사용자가 직접 작성한 작업 요청 | 사용자의 목표, 표현 방식, 우선순위가 담긴 데이터 |
| 사용자 결정 | 승인/반려/선택지 응답 | 사용자의 판단 기준과 선호를 보여주는 고품질 데이터 |
| 작업 결과 | AI가 생성한 문서, 코드 제안, 실행 로그 | 실제 작업 과정과 산출물의 근거 |
| 사용자 피드백 | 수정 요청, 반려 사유, 회고 | 반복 실수를 줄이기 위한 핵심 데이터 |
| 외부 도구 | GitHub issue, 문서, 메시지 등 | 작업 맥락을 보강하는 외부 데이터 |
| 초기 seed 문서 | ChatGPT, Claude 메모리, 이메일 history 등 | 사용자의 성향 등을 초기에 빠르게 확보 가능한 데이터 |

### 6.2 데이터 형태 및 포맷

| 데이터 종류 | 형태/포맷 | 예시 |
|---|---|---|
| 작업 요청 | 자연어 텍스트 | "결제 설정 화면의 모바일 UX를 확인해 주세요." |
| 결정 기록 | JSON + 텍스트 요약 | 질문, 선택지, 사용자의 답변, 답변 이유 |
| 지식 note | Markdown | 사용자 선호, 반복 패턴, 작업 회고 |
| 문서 데이터 | Markdown, TXT, PDF 변환 텍스트 | README, PRD, 운영 문서 |
| 로그/검증 결과 | 구조화 JSON + 텍스트 로그 | 테스트 결과, 실행 명령, 오류 메시지 |
| 임베딩 데이터 | PostgreSQL pgvector | note나 예시 문장을 벡터로 변환해 의미 기반 검색에 사용 |
| 지식 연결 데이터 | Node/Edge 형태 | 개념, 문서, 작업 간 연결 |

### 6.3 데이터 특성 및 주의사항

초기에는 고정 데이터셋이 없습니다.

이 서비스의 핵심은 사용자가 서비스를 쓰는 동안 데이터셋이 생기는 dynamic RAG입니다. 처음 가입한 workspace는 빈 지식 기반에서 시작하고, 사용자의 작업 지시와 피드백이 쌓이면서 개인화된 데이터셋이 만들어집니다.

주의해야 할 점은 다음과 같습니다.

- 사용자의 결정과 피드백에는 개인적 선호와 민감한 업무 맥락이 들어갈 수 있습니다.
- AI가 생성한 내용을 그대로 지식으로 확정하면 오류가 누적될 수 있으므로, 출처와 검증 상태를 함께 저장해야 합니다.
- 반복 패턴은 충분한 근거가 쌓인 뒤에만 강한 규칙처럼 사용해야 합니다.
- 데모용 데이터는 실제 개인 데이터를 쓰지 않고 공개 가능한 synthetic 문서로 구성하는 것이 적절합니다.

### 6.4 데이터가 쌓이는 방식

| 단계 | 데이터 | 예시 |
|---|---|---|
| Raw Input | 원본 입력 | 사용자 지시, GitHub issue, 문서 내용 |
| Knowledge Note | 처리된 지식 기록 | "사용자는 release note를 PR에 포함하길 원합니다." |
| Repeated Pattern | 반복되어 중요해진 패턴 | "검증 없는 완료는 허용하지 않습니다." |
| Decision Memory | 사용자가 과거에 내린 결정 | "보안 위험이 낮은 A안을 선택했습니다." |
| Evaluation Fixture | 평가용 고정 샘플 | 회귀 테스트용 질문과 기대 검색 결과 |

## 7. 프로젝트 내에서 구현하고 싶은 대표 피처

### 7.1 대표 피처: 개인화된 작업 메모리 RAG

사용자가 이전에 내린 결정, 반려한 이유, 선호하는 작업 방식, 검증 기준을 지식으로 저장한 뒤, 다음 작업에서 자동으로 검색해 프롬프트와 답변에 반영하는 기능입니다.

예를 들어 사용자가 과거에 "테스트가 없으면 완료로 보지 않겠습니다"라고 피드백했다면, 다음 작업에서 AI는 결과를 낼 때 테스트 여부를 반드시 확인하고 Delivery Report에 포함해야 합니다.

### 7.2 세부 기능

| 피처 | 설명 |
|---|---|
| 작업 요청 분석 | 사용자의 자연어 요청을 작업 유형과 필요한 산출물로 분류합니다. |
| 과거 결정 검색 | 유사한 상황에서 사용자가 어떤 선택을 했는지 검색합니다. |
| 사용자 피드백 저장 | 반려 사유와 수정 요청을 재사용 가능한 지식으로 저장합니다. |
| 반복 패턴 추출 | 여러 번 등장한 기준을 중요한 규칙 후보로 정리합니다. |
| 근거 포함 답변 | AI 답변에 어떤 과거 문서나 결정을 참고했는지 함께 보여줍니다. |
| 검증 체크리스트 생성 | 과거 피드백을 바탕으로 이번 작업의 확인 항목을 제안합니다. |

## 8. 성공 기준

| 기준 | 측정 방법 |
|---|---|
| 지식 누적 | 작업 후 지식 note와 반복 패턴이 생성되는지 확인합니다. |
| 검색 재사용 | 다음 작업 프롬프트에 관련 과거 결정이 포함되는지 확인합니다. |
| 반복 실수 감소 | 같은 유형의 반려 사유가 다음 작업의 검증 기준에 반영되는지 확인합니다. |
| 증거 기반 완료 | Delivery Report에 테스트, 명령, 파일, 링크가 표시되는지 확인합니다. |
| 사용자 개입 감소 | 검증된 작업 1건당 사용자의 직접 개입 시간을 측정합니다. |
